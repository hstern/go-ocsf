# OCSF sample-event fixtures

This directory holds JSON sample-event payloads the conformance
tests (`internal/conformance/`, lands in OCSF-23) consume. One
subdirectory per upstream schema version; the active set is
selected by the `ocsf.SchemaVersion` constant the library
ships.

## Vendoring policy

- **One directory per schema version.** Schema bumps land
  alongside their fixture updates in the same PR — the
  conformance tests assert byte-stable round-trip, so a
  schema-version change that affects the wire format would
  surface as a diff in either the generated types
  (`codegen-diff` CI gate) or here (`go test`).

- **Hand-curated, not bulk-imported.** Upstream OCSF doesn't
  ship a class-by-class collection of canonical sample events
  in a vendorable form — the official samples are generated
  on demand by the OCSF Server (`schema.ocsf.io`) and don't
  carry stable identifiers across schema versions. Rather
  than pin to an unstable feed, this project hand-curates
  fixtures covering the shapes the conformance tests need:
  minimal payloads (required-field-only), full payloads
  (optional + profile-bearing), and forward-compat shapes
  (unknown `class_uid` for the BaseEvent round-trip).

- **Coverage is additive, not exhaustive.** v0.1.0 ships
  enough fixtures to exercise the conformance properties
  (byte-stable round-trip, validate-pass on full samples,
  unmapped passthrough, BaseEvent fallback). Class-by-class
  expansion lands additively — when a consumer reports a
  parse / validate issue against a class without a fixture,
  add a fixture exercising the failure shape in the same PR.

- **Each fixture is a single complete OCSF event** as a JSON
  document. No newline at end of file; tests are byte-strict
  about input.

- **No edits to vendored content without a reviewer note.**
  When a fixture changes (e.g. a schema bump renames an
  attribute), the PR body explains the upstream change and
  the diff is reviewed alongside the schema bump.

## Why not just embed `ocsf/examples`?

The `ocsf/examples` upstream repo carries integration-platform
mappings (Spark, Splunk), raw-log datasets (Microsoft, Zeek),
and encoding format demonstrations — not the OCSF-JSON event
payloads the conformance test needs. The samples that schema.ocsf.io
serves are generated by the OCSF Server from the schema
itself; they're a fair starting point but they aren't tagged,
aren't fetched at build time (that would couple CI to a live
service), and don't surface the failure shapes we want
fixtures to cover. The hand-curated set is the right
trade-off for a library that wants reproducible conformance
tests.

## Layout

```
internal/specfixtures/
├── README.md            ← this file
└── v1.3.0/
    ├── iam/
    │   ├── authentication-minimal.json
    │   └── authentication-full.json
    ├── findings/
    │   └── detection_finding-minimal.json
    ├── system/
    │   └── file_activity-minimal.json
    └── future/
        └── unknown_class-9999.json   ← forward-compat / BaseEvent
```

The category directories mirror `events/<category>/` for
quick mental mapping; the `future/` directory carries
forward-compat fixtures whose `class_uid` the library
intentionally doesn't recognize (exercises the BaseEvent
fallback path).
