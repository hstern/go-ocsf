# Contributor conventions

These notes describe how `go-ocsf` is laid out and the conventions
contributors and AI coding assistants are expected to follow when
proposing changes. They complement, and do not replace, the standard Go
community conventions (Effective Go, `gofmt`, the Go style guide).

## Project shape

- **Module path**: `github.com/hstern/go-ocsf`. The library name is
  `go-ocsf` and the importable package root is `ocsf`.
- **Go version**: 1.26 or later.
- **Dependencies**: zero non-stdlib runtime dependencies. Test-only
  dependencies are evaluated case-by-case; default is still the stdlib
  `testing` package with table-driven tests.
- **Schema version tracked**: the constant `ocsf.SchemaVersion`. Bumping
  it is a library-minor release.

Layout (target shape ‚Äî populated phase-by-phase):

- root package `ocsf` ‚Äî `SchemaVersion`, `Event` interface, `BaseEvent`,
  `Parse`, registration, validation, time helpers, category enums.
- `events/<category>/` ‚Äî generated per-event-class structs.
- `objects/` ‚Äî generated per-object structs (`User`, `Process`,
  `Device`, `Metadata`, ‚Ä¶).
- `internal/schema/v<x.y.z>/` ‚Äî vendored upstream OCSF schema JSON.
- `internal/gen/` ‚Äî codegen tool (`go generate ./...`).
- `internal/specfixtures/v<x.y.z>/` ‚Äî vendored upstream sample events
  for conformance testing.

## Source-file header

Every `.go` file the library ships carries this two-line header before
the `package` declaration:

```go
// Copyright 2026 The go-ocsf Authors
// SPDX-License-Identifier: Apache-2.0
```

Generated files use the same header but additionally include the
standard `// Code generated by internal/gen; DO NOT EDIT.` marker on
the line above the copyright block, per the Go generated-file
convention.

Documentation files (`README.md`, `CHANGELOG.md`, this file), workflow
YAML, and Dockerfiles do not carry the per-file header ‚Äî they're not
source files and SPDX tags in those formats add noise without benefit.
The full license text lives in `LICENSE` at the repo root.

## Design philosophy

`go-ocsf` is a standards-implementing library, not an application.
Decisions skew toward:

- **Wire / schema fidelity over ergonomic shortcuts.** If the spec
  says X, the library round-trips X verbatim ‚Äî even when a
  Go-idiomatic shortcut would prefer Y.
- **Lenient unmarshal, strict marshal.** Postel's law: accept what the
  wire delivers; validate at the marshal boundary only. A separate
  opt-in `Validate(...)` exists for consumers that want stricter
  checks.
- **Byte-stable round-trip on open-extension fields.** Open-extension
  carriers are decoded as `json.RawMessage`, not `map[string]any`, so
  that re-marshaling produces byte-identical output.
- **Stdlib HTTP only.** No framework-specific glue. Anything HTTP-shaped
  works with stdlib `net/http`, `chi`, `gorilla/mux`, and similar
  without special-casing.

Forward-compatibility: unknown `class_uid` values parse into
`BaseEvent` rather than producing an error. The `unmapped` extension
carrier passes through verbatim.

## Codegen

The bulk of the library's surface (per-object and per-event-class
structs) is generated by `internal/gen/` from the vendored upstream
OCSF schema. Two rules contributors should know:

1. **Generated files are committed.** Consumers must not need to run
   codegen to use the library. A `codegen-diff` CI guard fails if the
   committed generated files drift from what the vendored schema
   produces.
2. **The schema is vendored, not fetched at build time.**
   Reproducibility over freshness ‚Äî schema-version bumps are explicit
   pull requests.

Do not edit generated files by hand. If you need to change generated
output, change the generator (`internal/gen/`) and re-run
`go generate ./...`.

## Commit conventions

- **Imperative present tense title**, ‚â§72 chars, no trailing period
  (`add Foo`, not `Added Foo.`).
- **Body explains the why.** What forced this change ‚Äî a spec clause,
  an interop scenario, a discovered bug. Wrap at ~72 chars.
- **No conventional-commits prefixes** (`feat:`, `fix:`, `chore:`).
  Plain English in the body.
- **Concrete references in the body**: spec section numbers (¬ß5.1),
  RFC numbers (RFC 7515), public PR / commit SHAs. Anything verifiable
  by a stranger.

## CI gates

Every pull request runs:

- `static` ‚Äî `gofmt`, `go vet`, `go mod tidy -diff`, build.
- `test` ‚Äî `go test -race -shuffle=on ./...`.
- `lint` ‚Äî `golangci-lint run ./...`.
- `codegen-diff` (from Phase 2 onward) ‚Äî `go generate ./...` produces
  zero diff against committed files.

A separate daily workflow runs `govulncheck` against the main branch.

All required checks must be green before merge; the main branch
requires up-to-date branches for merge.

## Reporting issues and contributing

Until the library reaches `v0.1.0`, please raise questions and report
issues by opening a GitHub issue on this repository. Substantive design
proposals are best discussed in an issue before opening a pull request.
